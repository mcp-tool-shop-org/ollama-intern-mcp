# v2.3.0 — per-call model override across LLM-backed atom tools

**Released:** 2026-05-11

v2.3.0 adds per-call model override across LLM-backed atom tools. Orchestrators that maintain receipt-backed model identity (e.g., research-os reviewer profiles, calibration evidence, model-comparison proofs) can now specify an explicit model per call while preserving the tier/profile default behavior for tools that omit the override. Tier-based timeouts and fallback semantics are preserved. Composite/brief tools continue to use tier defaults; orchestrators needing per-call control should use atom tools directly.

This is a non-breaking additive minor. All v2.2.0 callers continue working unchanged.

---

## What landed

The 8 LLM-backed atom tools accept an optional `model: string` input:

- `ollama_extract`
- `ollama_classify`
- `ollama_summarize_fast`
- `ollama_summarize_deep`
- `ollama_research`
- `ollama_corpus_answer`
- `ollama_chat`
- `ollama_code_citation`

When provided, the first attempt on the tool's tier runs against the caller-specified model instead of the tier-resolved default. Empty or whitespace-only inputs throw `ZodError` at schema parse — loud schema failure beats silent fallthrough.

Composite/brief/pack tools (`change_brief`, `incident_brief`, `repo_brief`, `repo_pack`, `change_pack`, `incident_pack`, `hypothesis_drill`, `triage_logs`) deliberately do NOT accept `model`. Their internal extract/classify calls continue to use tier defaults. This is the operator-contract distinction: atoms get per-call control, composites use tier defaults. Orchestrators that need per-call identity across composite work should call atoms directly.

## Schema additions

Tool input (every affected atom):

```ts
model: z.string().trim().min(1).optional()
  .describe(
    "Optional per-call model override. When provided, overrides the " +
    "tool's tier-resolved model for this call. The tier's timeout " +
    "(TIER_TIMEOUT_MS) still applies. On timeout, fallback uses the " +
    "tier-resolved model, NOT the override. Use for receipt-backed " +
    "orchestration that requires explicit model identity (e.g., " +
    "research-os reviewer profiles)."
  )
```

Envelope (added one new optional field):

```ts
/**
 * The model the CALLER asked for via input.model override. Present only
 * when override was supplied. Calibration-aware callers compare
 * model_requested vs model to detect fallback substitution.
 */
model_requested?: string;
```

The existing `model` field continues to mean "the model that actually produced this output," including post-fallback. The existing `fallback_from?: Tier` semantics are unchanged.

## Calibration fidelity — `model_requested` vs `model`

The two-field design is the load-bearing detail. A single field would have collapsed two different facts into one, hiding the corruption case that receipt-backed orchestrators care about:

```ts
if (env.model_requested && env.model !== env.model_requested) {
  // The tier-fallback path substituted a different model.
  // This is the case research-os reviewer receipts need to flag.
}
```

When `input.model` is omitted, `model_requested` is absent and the envelope shape is identical to v2.2.0 — callers that don't care about model identity see no schema change.

## Fallback semantics (locked)

1. `input.model` provided → first attempt uses that model. The tier's `TIER_TIMEOUT_MS` still applies.
2. On timeout → fallback to the tier-resolved fallback model (per the existing `TIER_FALLBACK` chain). **The fallback retry does NOT use `input.model`** — the override is dropped, and the cheaper tier resolves its own model. This keeps the degraded path predictable; the caller's chosen model may not fit the cheaper tier's role at all.
3. `input.model` omitted → existing tier/profile resolution unchanged. `model_requested` absent from envelope.
4. Empty / whitespace-only `input.model` → `ZodError` at schema parse, NOT silent fallthrough.

The integration point in each tool is a one-line change:

```ts
const model = input.model ?? resolveTier(tier, profile.tiers);
```

## Tests

762 total (up from 725 in v2.2.0). New coverage:

- **Per-tool tests** on all 8 atom tools (extract, classify, summarize_fast, summarize_deep, research, corpus_answer, chat, code_citation): override threading to the underlying Ollama call, tier-default fallthrough when omitted, `ZodError` on `""` and `"   "`.
- **Integration test** in `tests/regressions.test.ts` — the research-os calibration pattern: `frame + model` together on `ollama_classify` honors both (prompt contains the frame; underlying call uses the override; envelope surfaces `model_requested === model` with no `fallback_from`).
- **Fallback-semantics test** (the load-bearing one) in `tests/regressions.test.ts`: `input.model = "deepseek-coder:33b"` on `ollama_extract`, workhorse timeout forced to 20ms via the existing `timeouts` override seam. First attempt with the override times out, fallback to instant uses the tier-resolved model. Assertion checks both fields: `env.model_requested === "deepseek-coder:33b"`, `env.model === <instant-tier model>`, `env.fallback_from === "workhorse"`. A single-field assertion misses the corruption case.
- **`tests/version.test.ts`** — `VERSION === pkg.version`.

## Migration notes

Drop-in upgrade from v2.2.x. No breaking changes. Existing callers that do not pass `model` see identical behavior in every envelope field except for the unconditional absence of `model_requested`. New callers can begin specifying `model` per call where receipt-backed identity is required.

Recommended adoption posture for research-os v0.8.0 and similar orchestrators:

- Use the atom tools (`ollama_extract`, `ollama_classify`, `ollama_research`) for any work that needs receipt-backed model identity.
- Inspect `env.model_requested` and `env.model` on every envelope; treat `env.model_requested && env.model !== env.model_requested` as the substitution flag.
- Continue using composite/brief tools where tier defaults are acceptable — they will not surface `model_requested` because they do not accept the input.

## Bug fixes

- **`src/version.ts` drift.** The runtime `VERSION` constant was hardcoded `"2.0.0"` and silently drifted across v2.1.0 and v2.2.0 (both shipped reporting the stale identity string). Replaced with a direct import from `package.json` using `with { type: "json" }`. A new `tests/version.test.ts` locks `VERSION === pkg.version` so future bumps stay in sync automatically. tsconfig already had `resolveJsonModule: true` and `module: NodeNext`, so no config change was needed.

## What this unblocks

- **research-os v0.8.0** — reviewer profiles need to thread an exact model identity through MCP without losing the tier's timeout discipline. v2.3.0 is the minimum interface that satisfies that contract.
- **Calibration evidence** — receipt-backed comparisons across reviewer profiles (`hermes3:8b` single-pass vs `mistral-nemo:12b` two-pass etc.) now flow through MCP envelopes that record the requested vs actual model directly.
- **Model-comparison proofs** — A/B comparisons against the same atom-tool input can now be expressed as two separate calls with different `model` values, with envelopes that lock the identity assertion at receipt time.

## Out of scope (deliberate)

- No new tier types, no new profile fields, no new env vars.
- No changes to the `TIER_FALLBACK` chain.
- No `model` override on composite/brief/pack tools — atom tools only.
- No `timeout_ms` or `temperature` per-call overrides — separate proposal if/when needed.
- No widening into non-atom tools (`corpus_search`, `corpus_index`, `embed`, `embed_search`, `doctor`, `log_tail`, `code_map`, `batch_proof_check`, `artifact_*`, etc.).
- No new top-level tools.
