# v2.4.0 — per-tier `num_ctx` control on the profile system

**Released:** 2026-05-12

v2.4.0 adds per-tier `num_ctx` control to the profile system. Operators on VRAM-constrained hardware can now set smaller context windows for workhorse/instant tiers to prevent CPU spill while keeping deep-tier defaults. The `dev-rtx5080` profile now ships with `instant: 4096`, `workhorse: 8192`, `deep: unset` — sized to keep hermes3:8b in the RTX 5080's 16GB VRAM for fast tools.

This is a non-breaking additive minor. All v2.3.0 callers continue working unchanged.

---

## Why this release

The research-os v0.8.0 Phase 1 diagnostic surfaced a load-bearing operational gap: hermes3:8b at the default 32K context on RTX 5080 16GB VRAM spills to CPU. Workhorse-tier `ollama_extract` then turns into a 5-10× latency event — and in the dogfood case, started timing out outright. The root cause isn't the model or the tier discipline; it's that the MCP server had no way to express "this tier should run at a smaller context budget."

v2.4.0 fixes it at the profile layer. Operators tune by selecting / editing the profile; the runner threads the value to Ollama on every generate call routed to that tier.

## What landed

**Schema additions:**

```ts
// src/tiers.ts
export interface TierConfig {
  instant: string;
  workhorse: string;
  deep: string;
  embed: string;
  num_ctx?: {           // ← new
    instant?: number;
    workhorse?: number;
    deep?: number;
    embed?: number;
  };
}

// src/envelope.ts
export interface Envelope<T> {
  // ... existing fields ...
  num_ctx_used?: number;  // ← new — present only when sent
}
```

**Profile defaults:**

| Profile | instant | workhorse | deep | embed |
|---------|---------|-----------|------|-------|
| `dev-rtx5080` | 4096 | 8192 | UNSET | UNSET |
| `dev-rtx5080-qwen3` | 4096 | 8192 | UNSET | UNSET |
| `m5-max` | UNSET | UNSET | UNSET | UNSET |

UNSET is load-bearing: when a tier's `num_ctx` is not set, the MCP server omits the `num_ctx` field from `options` entirely and Ollama uses its model-loaded default. This is the v2.3.0 contract, preserved exactly.

**Threading:** every LLM-backed call routes through one of three paths — `runTool` (single-call atom + composite path), `runBatch` (batch-mode atom path), or `handleChat` (the non-runner chat tool). Each path resolves the active tier's `num_ctx` and includes it in `options` ONLY when set. Prewarm and `corpus_search` explain sub-calls use the same resolution so they don't trigger Ollama to reload at a different context.

## Locked semantics

1. **Profile's tier `num_ctx` set** → request carries `options.num_ctx = <value>`; envelope carries `num_ctx_used: <value>`.
2. **Profile's tier `num_ctx` unset** → request omits `num_ctx` entirely; envelope omits `num_ctx_used`. Ollama uses its model-loaded default. This is the v2.3.0 contract.
3. **Fallback retries** resolve `num_ctx` against the FALLBACK tier — same posture as `modelOverride`. A workhorse-then-instant fallback uses instant's `num_ctx` (or stays unset). The degradation contract stays per-tier-coherent.
4. **No per-call `num_ctx`** override on any tool input. Operators tune by selecting / editing the profile.

## Why no per-call override?

The release is deliberately profile-level only. Per-call `num_ctx` would push context-budget reasoning out of the operator's hands and into every caller's tool invocation. The dogfood case that motivated the release isn't "different calls need different contexts" — it's "this hardware has a VRAM ceiling that the profile system should encode." Per-tier on the profile is exactly the right granularity.

If a future call surfaces the case where per-call control matters (a single workhorse call needs to push past the profile's 8K), the pattern is open: add a `num_ctx?: number` input on the affected atom tools the same way `model` got added in v2.3.0. v2.4.0 stays disciplined on scope.

## Why no deep-tier change?

`deep` stays UNSET on every shipped profile. The motivating diagnostic is workhorse-tier (the spill happens on short-shape `ollama_extract` calls that should be fast, not on long-context briefs). Reducing deep's context budget without dogfood evidence would silently shrink the long-context surface that `ollama_research` and `ollama_summarize_deep` exist to serve. If deep-tier spill ever surfaces in a future diagnostic, the same plumbing covers it — flip a number in the profile.

## Tests

792 total (up from 762 in v2.3.0). New coverage:

- **`tests/numCtx.test.ts`** — the dedicated v2.4.0 file. Covers each of the 8 atom tools with a `{set + present}` and `{unset + absent}` pair, batch-mode positive/negative on `extract`, composite via `change_brief`, and a diagnostic regression smoke (`extract` on `dev-rtx5080` carries `num_ctx: 8192` end-to-end).
- **`tests/profiles.test.ts`** — per-profile `num_ctx` assertions: `dev-rtx5080` sets `instant=4096/workhorse=8192` and leaves `deep/embed` UNSET; `dev-rtx5080-qwen3` mirrors it; `m5-max` leaves everything UNSET. `loadProfile` carries the map through.
- **`tests/tiers.test.ts`** — `resolveNumCtx(tier, config)` helper: returns the value when set, `undefined` otherwise, `undefined` for every tier when the map itself is absent.
- **`tests/envelope.test.ts`** — `num_ctx_used` field shape: present in builder output only when `numCtxUsed` was passed to the builder.

The "absent when unset" assertion is what proves backward-compat — present on every atom tool's negative-case test.

## Migration notes

Drop-in upgrade from v2.3.x. No breaking changes. Existing callers see identical Ollama wire payloads unless they're on `dev-rtx5080` / `dev-rtx5080-qwen3` (where `instant`/`workhorse` calls will start carrying explicit `num_ctx`). The fix is the point — operators on those profiles get the spill-prevention behavior for free.

If you've been pinning a custom model via `INTERN_TIER_*` on a constrained-VRAM box and want the old 32K behavior, switch profile to `m5-max` (which leaves `num_ctx` unset on every tier) or edit the profile's `num_ctx` map directly.

## What this unblocks

- **research-os v0.8.0 Phase 1** — workhorse-tier `ollama_extract` calls on RTX 5080 stop spilling to CPU and stop timing out.
- **VRAM-constrained operators in general** — the profile system can now encode hardware ceilings on context budget, not just on model choice.
- **Future per-tier ops knobs** — the threading shape (resolve per active tier, set explicitly only when defined, surface on envelope) is the template for any future per-tier control if/when one becomes necessary.

## Out of scope (deliberate)

- No per-tier `num_gpu` / `num_thread` / batch size.
- No profile inheritance / composition.
- No runtime `/api/ps` querying for effective-context detection.
- No per-call `num_ctx` override on tool inputs (profile-level only).
- No deep-tier `num_ctx` change on shipped profiles.
- No new tools.
