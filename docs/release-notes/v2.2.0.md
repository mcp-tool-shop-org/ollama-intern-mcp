# v2.2.0 ‚Äî local evidence-worker role contract

**Released:** 2026-05-11

v2.2.0 ships the local evidence-worker role contract: frame-bound topicality outputs on `extract` / `classify` / `summarize`, structured abstention on `ollama_research` / `ollama_corpus_answer`, retrieval scores preserved through brief evidence, and citation line-range bounds-checking. Designed in response to a research-os product proof that surfaced relevance as a first-class missing concept. research-os v0.8.0 (in development) will consume this contract via MCP, replacing direct-Ollama call paths.

This is a non-breaking additive minor. All v2.1.0 callers continue working unchanged. New behavior unlocks only when new optional inputs are supplied.

---

## What landed

### Frame-bound extraction
`ollama_extract`, `ollama_classify`, `ollama_summarize_fast`, and `ollama_summarize_deep` accept an optional `frame: string` input ‚Äî the section purpose, question, or topic the call is being made FOR. When supplied, the model first judges whether the source addresses the frame, then either extracts/classifies/summarizes only the frame-relevant content or surfaces an explicit off-topic signal. Off-topic sources are flagged in a structured output field rather than paraphrased into the caller's schema.

Output fields (present only when `frame` was supplied):
- `extract`: `frame_alignment: { on_topic: boolean, reason: string, unaddressed_aspects?: string[] }`
- `classify`: `off_topic: boolean`, `off_topic_reason: string | null` (label is forced `null` when `off_topic: true`, regardless of which label fit best in isolation)
- `summarize_fast`: `on_topic: boolean | null`
- `summarize_deep`: `frame_addressed: boolean | null`, `unaddressed_sources: string[]`

### Structured abstention on `ollama_research`
`ollama_research` now distinguishes "answered with grounded citations" from "answered without grounding" from "abstained." New output fields:
- `weak: boolean` ‚Äî true when answer text is non-empty but `citations[]` is empty.
- `abstained: boolean` ‚Äî true when the model's prose matches an abstention pattern ("the sources do not contain", "insufficient information", "cannot answer", etc.); citations are stripped when this fires.
- `sources_address_question: boolean | null` ‚Äî explicit signal about retrieval / source relevance.

Empty `citations[]` paired with a non-empty `answer` is no longer silent success.

### Topicality threshold on `ollama_corpus_answer`
Optional `min_top_score: number` (0.0‚Äì1.0) input. When the top retrieval score is below this floor, the tool short-circuits with `abstained: true`, `citations: []`, an explanatory `answer`, and a coverage note. The model is never invoked. Closes the "five sub-threshold hits still drive a confident answer" failure mode that the v2.1.0 `weak: true` rule (which only fired on `hits.length < 2`) did not catch.

Each `CorpusAnswerCitation` now also includes `score: number` (the underlying retrieval score for that chunk), so operators can audit retrieval quality directly from the envelope.

### Retrieval score preservation
The `score` field on corpus retrieval hits now propagates through `corpusHitsToEvidence` to the `EvidenceItem` records consumed by `ollama_incident_brief`, `ollama_repo_brief`, and `ollama_change_brief`. Briefs synthesized from corpus material can now expose the underlying retrieval grounding rather than dropping it at the corpus‚Üíevidence boundary.

`assembleEvidence` accepts an optional `corpus_min_evidence_score: number` parameter, and the three brief tools surface it as a top-level input. Below-threshold corpus chunks are dropped from the assembled evidence with a counted `assembly_notes` entry that flows into the brief's `coverage_notes`.

### Citation `line_range` bounds-checking
`src/guardrails/citations.ts` now validates that each citation's `line_range` falls within the actual file's line count. Out-of-bounds ranges have the `line_range` dropped (the path is preserved as a path-only citation) and an envelope warning is emitted. Matches the existing posture on `ollama_code_citation`.

### Operator-contract docs corrected
- README cathedral example: `chunk_id` ‚Üí `chunk_index` (the actual return field).
- "Every claim in `answer` cites a chunk id validated server-side" rewritten to disclose that server validation is path/range-bound, not content-bound.
- README cathedral example annotated: `weak: false` means "‚â•2 evidence items assembled," not "hypotheses vetted."
- Evidence Laws section adds an explicit "ID-validated, not content-validated" bullet.
- `docs/marketing-research.md` "Evidence-first. No fiction." qualified inline.
- Handbook tool rows annotated with v2.2.0+ notes pointing at the new README sections.

## Schema additions (additive, non-breaking)

All new fields are optional inputs or conditional outputs. The v2.1.0 envelope shape is preserved when the new optional inputs are not used.

| Tool | New input | New output (when input supplied) |
|---|---|---|
| `ollama_extract` | `frame?: string` | `frame_alignment?: { on_topic, reason, unaddressed_aspects? }` |
| `ollama_classify` | `frame?: string` | `off_topic?: boolean`, `off_topic_reason?: string \| null` |
| `ollama_summarize_fast` | `frame?: string` | `on_topic?: boolean \| null` |
| `ollama_summarize_deep` | `frame?: string` (alongside existing `focus`) | `frame_addressed?: boolean \| null`, `unaddressed_sources?: string[]` |
| `ollama_research` | (none new) | `weak?: boolean`, `abstained?: boolean`, `sources_address_question?: boolean \| null` |
| `ollama_corpus_answer` | `min_top_score?: number` | `abstained: boolean`, per-citation `score: number` |
| `ollama_incident_brief` / `ollama_repo_brief` / `ollama_change_brief` | `corpus_min_evidence_score?: number` | Corpus `EvidenceItem` records now carry `score?: number` |

## Seed regression ‚Äî the verification that closes the loop

The slice's contract is verified against the literal research-os fresh-pack failure case: arxiv 2112.10422 ("Cosmological Standard Timers from Unstable Primordial Relics") fetched under the section-01 evidence-custody frame *"What does evidence custody mean in local-first vs cloud LLM deep-research workflows, and which produces a more inspectable evidence chain?"* ‚Äî the source pre-slice produced 15 cosmology claims that passed extraction with no off-topic signal.

The regression at [`tests/regressions.test.ts`](../../tests/regressions.test.ts) covers 9 assertions across all 5 affected tools, using the verbatim fixture text from [`E:/AI/local-first-vs-cloud-research/evidence/excerpts/src_f080819ad5c1.jsonl`](file:///E:/AI/local-first-vs-cloud-research/evidence/excerpts/src_f080819ad5c1.jsonl):
1. `extract` surfaces `frame_alignment.on_topic = false` on the cosmology source.
2. `extract` back-compat: same call without `frame` returns the legacy shape.
3. `classify` surfaces `off_topic = true`, label forced `null`.
4. `classify` back-compat: no `frame` ‚Üí no `off_topic` field.
5. `summarize_fast` surfaces `on_topic = false`.
6. `summarize_fast` back-compat.
7. `summarize_deep` surfaces `frame_addressed = false` + `unaddressed_sources` populated.
8. `summarize_deep` back-compat.
9. `corpus_answer` with `min_top_score` above retrieval scores ‚Üí `abstained: true`, model never invoked.

Tests are mocked-LLM contract tests (deterministic in CI) ‚Äî they verify schema-and-prompt wiring, not live model behavior. Live-model verification against the fresh pack will run as part of research-os v0.8.0 integration.

## Deferred items (captured at [`docs/reports/SWARM_REPORT_2026-05-11_role-contract.md`](../reports/SWARM_REPORT_2026-05-11_role-contract.md) "Slice outcome" section)

These items were identified by the 2026-05-11 role-contract dogfood but not landed in v2.2.0. They are candidates for v2.3.x:

- **Frame integration in brief tools** for `path` / `diff` / `log` inputs. The brief tools accept `corpus_min_evidence_score` (the corpus relevance gate) but do not have a topicality check on path-supplied or diff/log-supplied sources. Brief MCP descriptions tell operators to pre-filter via `extract --frame` before assembling source_paths; deeper integration is deferred.
- **Server-pulled `excerpt` field on `ResearchResult.citations`** ‚Äî would match `ollama_code_citation`'s shape. Bounds-check landed; server-pulled excerpt did not.
- **`repo_pack` source-path validation** on `OnboardingFacts.entrypoints[].file` / `.config_files[]` ‚Äî the only place in the codebase where a model-fabricated path becomes a stored fact without operator-visible friction.
- **Brief `claim_supported_by_excerpt` field** OR rename of `evidence_refs` to reflect "ref-bound, not content-bound." Claim-to-evidence semantic grounding remains honor-system in briefs.
- **`confidence_source: "model_self_report"` provenance tag** on `classify` and brief `confidence` fields.
- **Citation validator unification** ‚Äî five per-tool implementations (`research.ts`, `corpusAnswer.ts`, `codeCitation.ts`, `briefs/common.ts`, `repoPack.ts`) instead of one shared module.

## Migration notes for consumers

**No migration required.** v2.2.0 is a drop-in upgrade from v2.1.0.

If you want to use the new behavior:
- **Frame-bound extraction**: pass an optional `frame: string` argument to `ollama_extract` / `ollama_classify` / `ollama_summarize_fast` / `ollama_summarize_deep`. Surfaced fields appear only when `frame` is supplied; absent otherwise.
- **Topicality-thresholded corpus answers**: pass `min_top_score: number` (e.g. `0.35` for cosine retrieval). Below-threshold queries short-circuit with abstention.
- **Per-citation retrieval grounding in briefs**: pass `corpus_min_evidence_score` to `ollama_incident_brief` / `ollama_repo_brief` / `ollama_change_brief` to drop low-relevance corpus chunks before synthesis. The score is also present on every assembled `EvidenceItem` whose source is a corpus chunk.
- **Detect ungrounded research output**: check `result.weak` and `result.abstained` on `ollama_research` results before treating prose as authoritative.

**Calibration note**: existing reviewer profiles (`hermes-two-pass-deterministic`, `mistral-nemo-two-pass`) were calibrated against the pre-v2.2.0 prompts. When `frame` is omitted, prompts are identical to v2.1.0 and calibration receipts remain valid. The frame-aware branch is operational but not yet calibrated; live-model calibration of frame-aware extraction is deferred to a future profile pass.

## research-os v0.8.0 (in development)

research-os' README declares `ollama-intern-mcp` as a runtime requirement, but the current code path uses direct-Ollama call sites bypassing the MCP entirely. This is architecture drift, identified during the 2026-05-11 product proof. research-os v0.8.0 will:
- Add an MCP client path replacing the four direct-Ollama call sites (extract, review, contradict, discover).
- Pass section purpose into ollama-intern-mcp tools as `frame`.
- Respect the topicality and abstention outputs surfaced by this v2.2.0 contract.
- Rename or remove the internal `ollama-intern.ts` stub files.
- Update the README dependency declaration to match the actual runtime requirement.

research-os v0.7.1 is held pending v0.8.0 architecture recovery.
